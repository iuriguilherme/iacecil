Phase 1

Codebase Context

ia.cecil began as a runnable script, became a module, and is now an installable package. The packaging configuration carries a sediment layer from each era, and no two layers agree. The grounding below is drawn directly from the repository's packaging files.

The core contradiction
pyproject.toml declares build-backend = "setuptools.build_meta" but lists every dependency under [tool.poetry.dependencies] — tables setuptools never reads. The runtime dependencies in pyproject.toml are therefore inert. There is no PEP 621 [project] table at all.
Five dependency sources, none authoritative
[tool.poetry] (inert) · setup.py (auto-generated by the abandoned dephell tool; install_requires only) · setup.cfg (metadata + duplicated [options]/[files] with conflicting package_dir syntax) · Pipfile (the de-facto working one — pinned, with platform markers and dev-packages) · requirements.txt (fully pinned but stale — e.g. boto3==1.36.10 vs the Pipfile's 1.40.76).
Version is defined in four-plus places
pyproject.toml hardcodes 0.3.0; setup.cfg reads attr: iacecil._version.__version__; src/iacecil/__init__.py resolves a version at import time (latest git tag → _version.py → hardcoded '0.0.0' fallback); and _version.py holds yet another copy. They can and do disagree.
Locking is hand-synced
parse_lock.py is a loose root script whose only job is to read Pipfile.lock and print pins back out in Pipfile/poetry format — manual reconciliation between two lock representations, made physical. uv.lock is a 52-byte header with no packages: uv was reached for but never actually locked anything.
Run-ways & environment drift
Run commands live in the Pipfile's pipenv-only [scripts] table (test, prod, dev via uvicorn, paola) — not portable to other tools. There is no [project.scripts] console-script entry point, so the iacecil command only exists via python -m iacecil or pipenv run. The Dockfile (misspelled) pins python:3.10 while every other source requires 3.11.
Hygiene smells
install-requires — a junk PyPI package — is listed as a real dependency in three files. Stray root scripts (update_personas.py, update_personas2.py, test_sanitizer.py, parse_lock.py, ce-adversarial-reviewer.json) sit outside any package or tests/ directory.
Project shape (constraints to preserve)
src/ layout with two distribution packages, iacecil and plugins; Python 3.11; dynamic-plugin and pluggable-connector architecture (Telegram, Discord, Mastodon, Matrix, XMPP, Furhat). The four run-ways and tool-agnosticism are explicit operator requirements, not negotiable scope.

1

PEP 621 [project] table as the single source of truth

Axis · Metadata SoT Confidence · High (92%) Complexity · Medium Keystone
Description

Move all package metadata — name, description, authors, classifiers, URLs, requires-python, dependencies — into a single standard [project] table in pyproject.toml. Declare the version once with dynamic = ["version"] and let the build backend read it from the code (or from a git tag), collapsing the four current version sources into one. Then delete [tool.poetry], setup.py, and the metadata duplication in setup.cfg. PEP 621 is the format every modern tool — uv, poetry, pdm, hatch, pip — reads natively, so one table is simultaneously the manifest for all of them.

Basis

direct: pyproject.toml sets build-backend = "setuptools.build_meta" yet lists deps under [tool.poetry.dependencies] (setuptools ignores them). Version appears as 0.3.0 in pyproject, attr: iacecil._version.__version__ in setup.cfg, and a git-tag→_version.py→'0.0.0' cascade in src/iacecil/__init__.py. PEP 621 is the documented standard that supersedes all of these.

Rationale

This is the keystone every other idea hangs from. The operator's actual requirement — "tool-agnostic" — is PEP 621: tool-agnosticism is not achieved by supporting many config formats but by adopting the one format the PEPs made universal. Once the [project] table exists, a single edit propagates to uv, poetry, pip, and conda alike, and the version stops being able to disagree with itself.

Downsides

Touching the live install path for a running multi-bot deployment — needs a verified pip install -e . + import smoke test before the old files are deleted. Dynamic-version wiring depends on backend choice (Idea 5), so the two are best sequenced together.

2

Connectors as optional-dependency extras

Axis · Dependencies & extras Confidence · High (84%) Complexity · Medium
Description

Split the dependency list into a lean core plus [project.optional-dependencies] extras that mirror the connector architecture: telegram, discord, mastodon, matrix, xmpp, furhat, plus convenience groups like web and all. An operator running only Telegram installs iacecil[telegram] and skips Matrix's crypto stack, Discord's gateway library, and Furhat's robot SDK. The install surface becomes self-documenting.

Basis

reasoned: the codebase already treats connectors as independently loaded, config-gated modules (ConnectorManager loads by credentials; a connector failure marks itself down without killing siblings). Their dependencies are therefore already optional at runtime — making them optional at install time only aligns the package metadata with an architecture that exists. external: PEP 621 optional-dependencies is the standard mechanism, used widely (e.g. pandas[all], uvicorn[standard]).

Rationale

It compounds with the connector design: every new connector becomes one new extra, and the supported-platform list is legible straight from pyproject.toml. It also slims a bare pip install iacecil down to a core that can actually resolve quickly, which matters for the heavy Matrix/web3/ML dependencies currently forced on every install.

Downsides

Requires deciding what truly belongs in core vs an extra (some shared HTTP/async libs are genuinely cross-cutting). Mis-bucketing produces import errors at runtime instead of install time — needs a per-extra import smoke test in CI.

3

Generated-not-authored locking — commit exactly one lock

Axis · Locking & reproducibility Confidence · High (88%) Complexity · Low–Medium
Description

Adopt a clean manifest/lock split: pyproject.toml holds abstract version ranges (the only thing humans edit); exactly one committed lockfile holds the resolved pins; and any other pin format consumers need (a requirements.txt for plain-pip or Docker) is exported from that lock by a command, never hand-maintained. Delete parse_lock.py, Pipfile.lock, and the hand-edited requirements.txt as authoritative artifacts. Pick one lock tool to own resolution; everyone else consumes its export.

Basis

direct: parse_lock.py exists solely to transcribe Pipfile.lock into pin syntax by hand; requirements.txt has already drifted from the Pipfile (boto3==1.36.10 vs 1.40.76); uv.lock is an empty stub. external: the manifest/lock split is the settled norm across ecosystems (npm package.json + lockfile, Cargo, Bundler); uv export / pip-compile produce a requirements.txt from a lock as a build artifact.

Rationale

The current pain is two-directional hand-syncing between formats that inevitably drift — three lockfiles that already disagree. Designating one as generated truth and treating the rest as exports removes the entire class of "which pin file is right?" questions and lets a CI step (not a human) keep pip-users' file current.

Downsides

Picking the resolving tool brushes against the tool-agnostic goal — mitigated because the output (an exported requirements.txt) is universally consumable even if the resolver is one specific tool. The committed lock is only as reproducible as the operator's willingness to install from it.

4

Portable entry points + a tool-agnostic dev task runner

Axis · Run-ways & entry points Confidence · High (86%) Complexity · Low
Description

Deliver the four run-ways from one config. Add a [project.scripts] console entry point (iacecil = "iacecil.__main__:main") so the installed package exposes a real iacecil command — the "installed package" way — while python -m iacecil (module way), pip install -e . (local-package way), and the uvicorn import target src.iacecil...:app (script/ASGI way) all keep working from the same metadata. Then migrate the pipenv-only [scripts] shortcuts (test, dev, prod, paola) into a tool-neutral task runner — a Makefile or justfile — that any venv can invoke regardless of whether the operator chose uv, poetry, or plain pip.

Basis

direct: run commands currently live in the Pipfile's [scripts] (e.g. dev = 'uvicorn src.iacecil.controllers._iacecil.development:app --reload'), which only pipenv run understands; there is no [project.scripts] entry point, so no iacecil command exists after a plain install. reasoned: PEP 621 console-scripts + a Makefile are both tool-agnostic; the four run-ways are properties of import paths and entry points, not of any one workflow tool.

Rationale

This is the idea that directly satisfies the operator's stated personal preference — all four invocation styles, none coupled to pipenv. It removes the last hard dependency on pipenv-specific tooling and makes the developer commands runnable under any environment manager.

Downsides

A Makefile is weaker on native Windows (mitigated by justfile or documented one-line equivalents). Console-script entry points require the package to actually be installed, so the smoke test must cover the installed path, not just -m.

5

Pick one PEP 517 backend; treat it as an implementation detail

Axis · Build backend & discovery Confidence · High (82%) Complexity · Low
Description

Resolve the setuptools-vs-poetry contradiction by choosing exactly one PEP 517 build backend in [build-system] and configuring src-layout discovery for it. The contenders — setuptools (already declared; native src-layout + find), or hatchling (clean PEP 621-first, simple dynamic-version plugin) — are both fully tool-agnostic: the backend is what python -m build / pip invoke under the hood, invisible to whether the operator drives with uv, poetry, or conda. The point is to declare one and delete the half-configured alternatives.

Basis

direct: [build-system].build-backend = "setuptools.build_meta" coexists with poetry dependency tables and a dephell-generated setup.py — three backends' worth of config, no single owner. reasoned: the build backend is the one packaging choice that genuinely is singular (you cannot build a wheel with two backends), and it is also the choice most invisible to operators, so it is the safe place to commit to one tool without violating the agnostic goal.

Rationale

Idea 1's dynamic-version wiring needs a definite backend to hang from, so this unblocks the keystone. It also reframes the operator's worry: "tool-agnostic" applies to the workflow layer (uv/poetry/pip), not the build backend — separating those two layers dissolves the apparent tension.

Downsides

Switching away from setuptools (if hatchling is chosen) is a small migration of the package_dir/packages discovery for the dual iacecil+plugins layout. Staying on setuptools is lower-churn but keeps a slightly more verbose discovery config.

6

Manifest & environment hygiene sweep

Axis · Dependencies / Run-ways Confidence · High (90%) Complexity · Low
Description

A correctness-and-consistency pass that ships alongside the structural moves: (a) remove the bogus install-requires dependency from all three files it appears in; (b) pin Python once in requires-python and reference it from the Dockerfile and any CI matrix — and fix the Dockfile name and its wrong python:3.10 base; (c) sweep the stray root-level scripts (parse_lock.py, update_personas*.py, test_sanitizer.py, ce-adversarial-reviewer.json) into a tests/, scripts/, or deletion as appropriate so the repo root is just packaging + docs.

Basis

direct: install-requires = "==0.3.0" appears in the Pipfile and as 'install-requires' in setup.py/pyproject — a real but junk PyPI package, a supply-chain smell. Dockfile reads FROM python:3.10 while requires-python >= 3.11 everywhere else. Loose root scripts (test_sanitizer.py, update_personas2.py) sit outside any package or tests/ dir.

Rationale

These are low-effort, high-certainty fixes that remove an outright install of a wrong package, a Docker build that runs the wrong interpreter, and root-level clutter that obscures which files are actually the packaging config. Bundled here so the structural ideas land on a clean surface rather than re-touching the same files twice.

Downsides

Mostly mechanical, but the stray scripts may be referenced by undocumented workflows — each move needs a quick grep for callers before relocating or deleting. Genuinely tactical: it warrants discussion only as the cleanup rider on the larger consolidation, not on its own.
