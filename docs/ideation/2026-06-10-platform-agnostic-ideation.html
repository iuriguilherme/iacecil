Platform-Agnostic ia.cecil

Date: 2026-06-10 Topic: platform-agnostic Mode: repo-grounded

Focus: The current state of the software is not agnostic as originally intended — it only works as a Telegram bot. We need to introduce new platforms like Discord and XMPP and let the bot configuration decide which platforms it connects to.

49 raw ideas
6 frames
7 survivors
5/5 axes covered

Grounding Context

Codebase Context. Python 3.10, aiogram pinned ==2.25.1 (EOL), plugin + personality architecture, Quart ASGI wrapper, ZODB persistence, per-bot pydantic configs in instance/bots/. The Telegram coupling is total: all ~30 plugins import aiogram directly (@dispatcher.message_handler, types.Message); IACecilBot subclasses aiogram's Bot with Telegram-specific send_* overrides and exception taxonomy (RetryAfter, BotKicked → sys.exit); personalidades receive types.Message; callbacks.py persists live aiogram objects to ZODB. The discord config dict exists with zero consuming code. Furhat support lives as a parallel controller tree (furhat_bot/, ~300 lines) duplicating personality logic.

Leverage points. Uniform plugin contract async def add_handlers(dispatcher); personalities are mostly text generation; flexible per-bot config; message logging centralized in one file. Strategy (STRATEGY.md): "Composability via runtime-loadable modules — plugins define capabilities, personalities define behavior — so any combination of platform, community, and experiment can be assembled without forking code."

External Context. Errbot's backend ABC with normalized Message + extras escape hatch; opsdroid's Connector ABC (connect/listen/send) with @register_event type-level capability detection; Matterbridge's transparent envelope whose Extra map the core never inspects; Hubot's adapter pattern with its known unsolved problem — cross-platform user identity; MS Bot Framework's Activity schema with channelData. Capability-negotiation analogies: XMPP XEP-0030, LSP handshake, GStreamer caps, Unreal RHI feature levels. Library landscape 2026: hikari is the cleanest Discord lib to wrap; slixmpp is the live XMPP choice (aioxmpp is dead); matrix-nio is sans-I/O. The aiogram 2→3 migration is itself breaking — its Router primitive decouples handler registration from the Dispatcher.

Past Learnings. No docs/solutions/ knowledge base exists; the in-repo prior art is the Furhat fork and the inert Discord config — the de facto record of past platform-integration attempts.

Topic Axes

  • connector-boundary — the adapter/driver interface each platform implements
  • message-model — normalized message/event types and platform escape hatches
  • plugin-migration — moving ~30 aiogram-coupled plugins to a new contract without a big-bang rewrite
  • config-and-selection — per-bot platform selection, multi-platform identity, instance/ config shape
  • capability-handling — platform-specific features without lowest-common-denominator dumbing-down

1

Aiogram compat shim — plugins run unmodified

plugin-migration Confidence 85% Complexity Medium
Description The plugins use a tiny, uniform slice of aiogram — roughly ten symbols: Dispatcher.message_handler, register_message_handler, types.Message.reply/.answer, filter kwargs, message.chat.type. Build iacecil.compat: a duck-typed emulation of exactly that surface, backed by a platform-neutral event bus instead of Telegram. All ~30 plugins run unchanged on any platform from day one; native migration happens opportunistically, plugin by plugin, and the shim is deleted last. A stronger variant promotes the Telegram Message schema itself to the canonical wire format, with connectors translating Discord/XMPP events into Telegram-shaped objects.
Basis direct: the entire plugin surface is two registration idioms across the whole tree — e.g. src/plugins/echo.py:37-38 is just @dispatcher.message_handler() + message.answer(); fediboard.py:40-48 adds only filters.Text and message.reply. external: FreeBSD's Linuxulator (emulate the used subset of a foreign ABI, never recompile consumers), Matterbridge's envelope, MS Bot Framework's channelData.
Rationale This idea emerged independently from all six ideation frames — the strongest convergence signal of the run. The migration cost everyone fears (30 plugin rewrites) is an illusion: emulating ~10 symbols converts a months-long port into one compatibility module plus per-platform connectors. It also absorbs the unavoidable aiogram 2→3 migration: since 2.25.1 is EOL and every plugin breaks under aiogram 3 anyway, the shim means that forced rewrite is paid once, not twice. A libcst codemod (rejected as a standalone idea) complements this for the eventual native migration.
Downsides The shim freezes a 2021-era API as the de facto plugin contract; Telegram-isms (chat.type, numeric chat IDs) leak into the abstraction's vocabulary. Needs a deliberate sunset plan or it becomes permanent.
2

Loopback connector as platform #0

connector-boundary Confidence 90% Complexity Low
Description Ship an in-tree stdin/REPL connector and define the acceptance test for the whole effort as: the framework boots, loads plugins and personalities, and holds a full conversation with no network platform at all. Telegram is then re-added as just another connector passing the same suite. Sequencing follows: loopback → compat shim → Furhat port (the in-house stress test — speech, no chat_id, no markdown breaks every Telegram assumption Discord wouldn't) → Discord/XMPP.
Basis reasoned: any plugin that cannot run against a terminal has, by construction, a hidden Telegram dependency the abstraction failed to capture — the flip turns "is it agnostic?" from an opinion into a runnable falsifier. direct: CLAUDE.md: "No test suite beyond running the bot itself."
Rationale One artifact buys three things: the definition-of-done for agnosticism, the project's first test harness (CI becomes possible — directly serving STRATEGY.md's "experiment completion rate" metric), and the development loop for every future connector author. It is the cheapest item on this list and de-risks everything else.
Downsides A terminal can't exercise media flows (voice, stickers, photos) realistically, so loopback alone under-tests the capability surface — it needs idea 7's degradation ladders to be honest about media.
3

Minimal connector ABC; connectors loaded like plugins

connector-boundary Confidence 80% Complexity High
Description Demote Telegram from chassis to connector #1. Define a small connector interface (connect/listen/send + event callbacks, after opsdroid), where each connector owns its own delivery policy — the retry, 2048-char message splitting, and dead-chat handling currently hardcoded in IACecilBot.exception_handler move inside the Telegram connector, and Discord/XMPP connectors bring their own. Connectors are discovered by name from config exactly like plugins are today, optionally via Python entry points (iacecil.connectors) so pip install iacecil-connector-xmpp adds a platform with zero core edits. The first work item inside this boundary: decouple operator observability — today log.py sends debug/error reports to a Telegram chat and bot.py calls sys.exit when kicked from the logger group, meaning the debug tool only works on the platform that's broken.
Basis direct: bot.py:55-262 holds the Telegram failure policy (flood-control retry, splitting at limit = 2048, BotKicked → sys.exit) inside a Bot subclass the whole framework instantiates; the dynamic-loading machinery already exists and is battle-tested at aiogram_bot/__init__.py:104 (import_module('.' + plugin, 'plugins')). external: opsdroid's three-method Connector ABC; Errbot backends; pytest/Datasette entry-points discovery.
Rationale Reuses the project's strongest proven mechanism (config-driven dynamic module loading) for its weakest area (platform coupling) — extending STRATEGY.md's composability bet one level down. Entry points convert the connector boundary from internal API to ecosystem API: community connectors can exist outside the repo. Without this, every future platform costs "fork the controller tree" — the Furhat tax, already paid once.
Downsides The largest single piece of work on this list; the ABC's verb set is a one-shot interface decision that every connector marries. Multi-connector lifecycle (N listeners on one event loop, partial failures) is genuinely new complexity the current single-dispatcher runner never faced.
4

Per-bot connections list; kill CLI mode dispatch

config-and-selection Confidence 85% Complexity Medium
Description Redefine a bot as persona + plugin set + connections, where connections is a list of (platform, credentials, per-platform overrides) entries in BotConfig. One unified runner reads the config and instantiates one connector per connection — the same bot identity rides Telegram, Discord, and XMPP simultaneously. The four-way CLI mode dispatch in __main__.py (production / fpersonas / furhatgpt / default) collapses: platform selection moves from "which runner module you launch" to data. The dead discord config dict becomes live the day a connector exists.
Basis direct: config.py:419 carries discord: dict = {'token': ""} with zero consuming code; config.py:162-172 types telegram/discord/furhat as parallel dicts on the same BotConfig — the schema already gestures at multi-platform while the runtime only ever reads telegram; __main__.py dispatches to four separate runner modules.
Rationale This is the user's literal acceptance criterion — "bot config decides which platforms it connects to" — made a first-class config primitive instead of an emergent property of which controller you launch. It also forces the runner unification all other ideas depend on.
Downsides Breaks the instance/bots/*.py schema (one migration for existing deployments) and deletes a CLI contract production systems currently invoke — needs a deprecation path for the systemd units that call pipenv run prod and fpersonas.
5

Personalities go platform-blind first

plugin-migration Confidence 80% Complexity Low–Medium
Description Personalities are nearly pure text generation, yet each ships its own add_handlers(dispatcher) touching aiogram directly. Strip that privilege: a personality becomes a declarative command→async-text-function map ({"start": start, "help": help, ...}), and the core auto-generates platform-appropriate registrations per connector. All 11 personality modules become platform-agnostic in one stroke without editing their text logic — and the Furhat controller's ~300 lines of duplicated personality handling collapse into the same map. The same persona speaks on Telegram, Discord, and a robot face months before plugin parity.
Basis direct: the personalidade layer is "mostly text generation" yet every module exposes add_handlers(dispatcher) receiving aiogram types; furhat_bot/ (personas/, borogodo.py, chatgpt.py) duplicates personality dispatch precisely because the aiogram entanglement made sharing impossible — the fork is the in-repo proof of what the current contract costs.
Rationale The cheapest large win available: the layer closest to pure data is coupled for no functional reason. Decoupling it makes "any personality on any platform" true immediately — the core promise of STRATEGY.md's composability claim — and ships a visible cross-platform result early, which keeps a long migration motivated.
Downsides Changes the authoring contract for all 11 personalities and every future one (an API freeze decision); personalities that do more than text (Furhat gestures, LED control) need the capability escape hatch from idea 6 to avoid re-coupling.
6

One data-model decision: envelope, persisted record, identity

message-model Confidence 65% Complexity High
Description Three things that look separate are one schema decision, and should be designed together. (a) The runtime envelope: a normalized core (text, sender ref, conversation ref, reply target, tags) plus a raw/extra escape hatch the core never inspects — Matterbridge's transparent-envelope pattern. (b) The persisted record: today callbacks.py does setattr(message, 'tags', ...) on live aiogram objects and hands them to zodb_logger — ZODB pickles aiogram class instances, meaning the database itself is aiogram-shaped and years of history become unreadable the day aiogram is upgraded or removed. Store the neutral record instead, with a one-time migration. (c) The canonical identity registry: a Person record keyed in ZODB, with each connector resolving its native ID (Telegram user_id, XMPP JID, Discord snowflake) via a link/claim flow — so allowlists, admin rights, and conversation memory follow the human across platforms.
Basis direct: callbacks.py:37-52 — every handled message flows through message_callback, which mutates the aiogram object before zodb_logger(message); the DB schema is aiogram, a silent lock-in worse than the import coupling. external: Matterbridge Extra / MS channelData escape hatches; Hubot's cross-platform identity gap is the named unsolved problem in all surveyed prior art; telephony's number-portability database is the industrial solution to the same shape.
Rationale The persisted-aiogram discovery is the sharpest finding of the grounding pass — a coupling that would bite any migration plan if found late. Identity is the part of multi-platform that retrofits worst: every plugin storing per-user state hardcodes Telegram IDs today. Solving envelope + storage + identity as one design makes the abstraction durable instead of cosmetic, and would be ia.cecil's actual differentiator over the prior art.
Downsides Schema decisions are near-irreversible in ZODB and this one spans persistence, security (who is an admin), and privacy (linking identities across platforms). Requires a data migration for existing instances. Confidence is moderate because the identity-linking UX (claim flow) is unvalidated.
7

Capability tiers and degradation ladders

capability-handling Confidence 60% Complexity Medium
Description Each connector advertises a capability set (voice, stickers, edits, markup dialect, max message length); the framework — not the plugins — owns degradation ladders applied automatically: sticker→image→emoji→text, voice→audio file→transcript, inline keyboard→numbered text menu, long message→split. Policy: degrade, never gate — no plugin may ever be platform-gated, so a text-only XMPP connector still runs the tts and qr plugins, just degraded. The closed five-verb output surface already enumerated in bot.py (send_photo/voice/video/message/sticker) is exactly where the ladders attach.
Basis direct: bot.py:264-300 — the entire output capability surface is five overridden methods in one file, a closed, ladder-able set. external: Unreal's RHI feature levels (the engine emulates missing GPU features downward, not just reports them — a step beyond XEP-0030/LSP capability reporting that none of the surveyed bot frameworks took); GStreamer caps negotiation. reasoned: the Furhat fork is the in-repo proof that capability mismatch forces forks when the framework doesn't own degradation.
Rationale Capability mismatch is why "agnostic" frameworks rot into lowest-common-denominator text pipes or per-platform forks. Framework-owned degradation is the third option: N+M code instead of N×M branches, written once in the core and inherited by every current and future plugin on every platform.
Downsides Degradation quality is judgment-heavy (a transcribed voice note is not a voice note); some fallbacks need real work (transcription). Risk of over-engineering before a second platform exists — sensible to ship the advertisement mechanism first and add ladders on demand.

Round 2 Grounding Context

What shipped since round 1. Ideas 2, 3, 5 and parts of 6 are now real: a four-verb BaseConnector ABC (connect/listen/send/disconnect), a ConnectorManager with command registry and per-connector plugin loading, loopback and xmpp connectors, the frozen Envelope dataclass with raw/extra escape hatches, neutral persistence (people.fs Person registry, messages.fs normalized records), and the strangler-fig arbitration rule — Telegram envelopes persist but legacy aiogram handlers own Telegram replies (documented in docs/solutions/architecture-patterns/strangler-fig-dispatch-arbitration.md).

Prepared but inert. The manager's allowlist already contains discord (connectors/__init__.py:74), _is_active already checks a discord token (:59), config.py:419 carries the discord config stub, and echo.py:56 already exports add_handlers_discord — only the connector module and a dependency are missing.

Remaining couplings (the round's targets). controllers/log.py is wholly Telegram: debug_logger/exception_logger/info_logger use Dispatcher.get_current() and bot.send_message to telegram.users.special.{debug,info} chat ids with aiogram markdown; tecido_logger is dead code that awaits blocking socket calls; zodb_logger still hands live aiogram objects to legacy zodb_orm. The neutral message record has no timestamp and no native message id, and dispatch() persists inbound only — outbound replies are never stored. production.py builds quart + aiogram and never instantiates ConnectorManager, so a configured xmpp connector simply never runs in production. Adding any new platform requires hand-editing the manager's allowlist and _is_active rules, and every plugin must alias add_handlers_<connector> per platform.

External context (carried from round 1 + cutoff knowledge). matrix-nio is the maintained async Matrix client; the Matrix client-server API's /sync long-poll is plain JSON over HTTP (aiohttp is already pinned). discord.py 2.x, hikari, and disnake are the live Discord options; message-content intent must be enabled in the developer portal. Platform message limits: Telegram 4096, Discord 2000, XMPP server-dependent (~4000 used), Matrix bounded by the 64 KiB event size.

Discord connector — three layers already prepared

new-connectors Confidence 90% Complexity Low
Description Create connectors/discord.py implementing the four-verb ABC: connect logs in the client, listen runs the gateway, send maps conversation_ref to a channel and reply_ref to a message reference, and the on_message event builds an Envelope (skipping the bot's own messages, mirroring the xmpp connector's self-echo guard). The library choice — discord.py, hikari (round 1's research pick), or disnake — is a brainstorm-level decision; all three fit the same connector shape.
Basis direct: the manager allowlist already contains discord and _is_active already checks its token (connectors/__init__.py:59,74); config.py:419 carries the config stub; echo.py:56 already exports add_handlers_discord. Only the module and the dependency are missing.
Rationale The cheapest possible second native platform — three separate layers were prepared for it and have sat inert since round 1 flagged the dead config dict. A first echo reply on Discord is this round's fastest visible win and the first real test of the manager's multi-connector lifecycle beyond loopback + xmpp.
Downsides A heavyweight new dependency in an already large Pipfile; the message-content intent must be enabled in the Discord developer portal (onboarding friction worth documenting); the 2000-character limit needs the chunking treatment idea 7 codifies.
2

Matrix connector — the designated architecture stress test

new-connectors Confidence 85% Complexity Medium
Description Create connectors/matrix.py speaking the Matrix client-server API. Two implementation variants for brainstorm: matrix-nio (maintained async client; E2E optional via nio[e2e] + olm) or a raw /sync long-poll loop over the already-pinned aiohttp — zero new dependencies; the unencrypted subset (sync, m.room.message events, send, join) is small. Unlike discord, matrix is not in the manager's allowlist or _is_active rules — adding it forces hand-editing both, which is exactly the point: it is the first concrete proof those hardcoded rules don't scale, feeding idea 7.
Basis direct: _load_connectors hardcodes the allowlist set and _is_active hardcodes per-platform credential rules (connectors/__init__.py:50-61,74) — every genuinely new platform requires manager edits. external: the Matrix C-S API is plain JSON long-polling; matrix-nio is the live async client per round-1 research; Matrix's federation and bridge ecosystem means one connector reaches many networks.
Rationale Matrix is federated and bridge-rich — strategically the highest-reach platform per connector written. And as the first platform the architecture did not anticipate, it is the designated stress test of this validation round: every rough edge it hits is a finding.
Downsides Login flow (token vs password) and sync-token persistence (where does next_batch live between restarts?) need decisions; E2E-encrypted rooms are silently unreadable without the olm stack; the raw-API variant means owning protocol code the library would otherwise own.
3

Connector-native runner (python -m iacecil connectors)

runner-entrypoint Confidence 90% Complexity Medium
Description A new runner module (e.g. controllers/_iacecil/connectors_runner.py) plus a connectors branch in __main__.py. It loads instance/_bots.py and per-bot configs exactly like production does, builds one ConnectorManager per bot, and runs them all under asyncio — no Quart, no legacy aiogram wrapper. production.py and testing.py are untouched. Optionally registers a built-in /status command listing each connector's up/down state, surfacing what the manager today only logs when it marks a connector down.
Basis direct: production.py builds quart_startup(aiogram_startup(...)) and never instantiates ConnectorManager — a configured xmpp section simply never runs in production; testing.py hardcodes DefaultBotConfig + loopback, so no instance-config path exercises real connectors either.
Rationale The user's stated requirement, and the keystone of the round: matrix, discord, and the log sinks are only provable in a long-running multi-connector process, which today does not exist outside the loopback REPL. Multi-bot parity with production from day one (same instance/_bots.py contract) avoids a second migration later, and the runner becomes the eventual production successor once the architecture is validated.
Downsides A third runner whose config-loading duplicates production's until a shared loader is extracted; polling only (no webhook/quart surface) — acceptable for a validation round, a known gap for real cutover.
4

Platform-neutral log sinks + ConnectorLogHandler

logging-pipeline Confidence 80% Complexity Medium
Description A new bot-config field log_sinks: [{platform, conversation_ref, level, tags}] and a stdlib logging.Handler subclass that formats matching records into Envelopes and delivers them through manager.send — so the operator channel can be a Matrix room, an XMPP JID, or a Telegram chat, chosen per bot in config. The Telegram-only debug_logger/exception_logger/info_logger remain for the legacy aiogram path but the connector path never imports them; the dead tecido_logger (awaits blocking socket calls — crashes if ever called) is deleted as part of the cleanup.
Basis direct: controllers/log.py:108-201 — operator logging requires Dispatcher.get_current() and sends via bot.send_message to telegram.users.special.{debug,info} chat ids with aiogram MarkdownV2 helpers; round 1 already flagged that "the debug tool only works on the platform that's broken". tecido_logger at log.py:59-106 awaits non-awaitable socket methods — dead on arrival.
Rationale Hooking the stdlib logging tree means the decoupling is free at every call site — no caller changes, and the manager's existing logger.error("Connector {name} failed") lines become operator alerts on whichever platform still works. This is the round's stated logging requirement made config, not code.
Downsides Re-entrancy is the sharp edge: a send failure inside the handler must never log through the same handler (recursion guard mandatory). Logging is synchronous while delivery is async — needs a queue or call_soon_threadsafe bridge. Error storms need rate limiting before this is left on in production.

Neutral persistence v2 — timestamps, native ids, outbound

persistence Confidence 80% Complexity Low–Medium
Description Complete the neutral message record: UTC timestamp, native message id, and direction (inbound/outbound) — populated from the Envelope (new optional fields or an extra convention). Persist outbound replies, which today vanish: dispatch() stores the inbound envelope, calls the handler, and sends the reply unrecorded. The legacy zodb_orm path stays read-only; zodb_logger's habit of pickling live aiogram objects gets no new callers.
Basis direct: persistence/neutral.py:89-96 — the record dict carries platform, refs, text, and tags but no timestamp and no native id, so stored history cannot answer "when" or "which message"; connectors/__init__.py:130-140 builds and sends the reply envelope without persisting it.
Rationale The stated persistence weakness, made concrete: the echo experiment's own round-trip cannot be verified from storage today. Timestamps and direction are the minimum substrate for any future conversation-memory or analytics feature, and capturing outbound now means the echo round produces complete conversational records from day one.
Downsides Schema evolution in schemaless ZODB dicts — readers must tolerate old records lacking the new fields; native-id capture requires each connector to thread its platform id into the envelope, a small per-connector tax.
6

Envelope middleware pipeline

dispatch-plugin-contract Confidence 70% Complexity Medium
Description dispatch() stops inlining its side effects: the ConnectorManager holds an ordered list of envelope observers (persist → log-sink tap → registry dispatch), and outbound replies pass back through the same chain — which is also where idea 5's outbound persistence naturally attaches. Each concern becomes a small, separately testable unit instead of another block inside one growing method.
Basis reasoned: dispatch() already imports persistence inside the function body (connectors/__init__.py:101), and this round adds log sinks and outbound persistence — three side-effect concerns accreting in one method is the trend line. external: ASGI middleware and aiogram's own middleware stack are the established shape for exactly this decomposition.
Rationale Makes the round's two decouplings structural rather than ad hoc: log sinks and persistence v2 land as middlewares instead of more inline code, and the next concern (metrics, rate limiting, filtering) costs a list entry instead of a dispatch() edit.
Downsides An abstraction tax on a codebase already mid-strangler-fig; ordering and failure semantics (does a persist failure block dispatch? — today it doesn't) must be made explicit; over-engineering risk if the round only ever needs two observers.
7

Contract v2 + echo conformance suite

dispatch-plugin-contract Confidence 70% Complexity Medium
Description Three contract fixes the new connectors will expose. (a) Connectors declare their own activation — a required_keys/is_active(conf) classmethod on each Connector class — and the manager discovers modules in connectors/ instead of maintaining a hardcoded allowlist plus per-platform credential rules. (b) Plugins expose one add_envelope_handlers(manager) entry point instead of aliasing add_handlers_<connector> per platform (the legacy Telegram aiogram entry stays). (c) A parametrized pytest conformance suite every connector must pass: envelope construction from a fake native event, send chunking at the platform's limit (Telegram 4096 / Discord 2000 / XMPP ~4000 / Matrix event size), self-echo guard, and an echo round-trip through a stub manager.
Basis direct: echo.py:54-56 — add_handlers_xmpp = add_handlers_loopback = add_handlers_discord = _add_connector_handlers is the aliasing smell, O(plugins × platforms) as both grow; _is_active and the allowlist (connectors/__init__.py:50-61,74) are the two places every new platform must hand-edit; chunking currently lives ad hoc inside xmpp.py:78 while telegram.py's connector send has none. reasoned: the round's purpose is finding weaknesses — codifying the echo exercise as contract tests means every future connector inherits the validation for free instead of repeating the manual experiment.
Rationale Turns this round's one-off experiment into a permanent regression net, and removes the two hand-edit chokepoints that ideas 1 and 2 will hit in practice. This is the compounding move: matrix and discord pay its cost once, every later connector rides free.
Downsides Changes the plugin entry contract while only echo uses the connector path — cheap now, but a decision every future plugin inherits; discovery-by-scan must skip base/__init__ and survive import errors gracefully.
